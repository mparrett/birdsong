# Repository Restructure Plan

## Summary

The repository should be recentered on `birdsong.py` as the single supported
root entrypoint. Experiments should remain available, but they should be moved
out of the main product path and grouped by status so active research stays
visible and old branches stop creating ambiguity. Challenge and coursework
material should be preserved in an explicit archive area rather than mixed into
the day-to-day working surface.

This plan favors deletion, consolidation, and truthful documentation over new
layers or abstractions.

## Main Drivers

- `README.md` has stale claims, dead references, and an overly optimistic
  support story.
- `justfile` still points key default commands at `poc.py` instead of
  `birdsong.py`.
- `test_bit_conversion.py` is stale and imports symbols that no longer exist.
- Historical clutter such as `DEV_NOTES.md`, the current `challenge/` area, and
  superseded experiment variants make the root harder to trust.

## Target Structure

Keep at repository root:

- `birdsong.py`
- `README.md`
- `justfile`
- `pyproject.toml`
- `docs/`
- `tests/`
- `tools/`
- `experiments/`
- `archive/`

Planned grouping:

- `tools/` for spectrogram, debug, and audio helper scripts that are not the
  core modem.
- `experiments/active/` for a small curated set of active research variants.
- `experiments/archive/` for superseded or duplicate experiment branches.
- `archive/challenge/` for coursework, challenge, and historical context that
  should be preserved but not treated as current product surface.

## Planned Curation

Active experiment set:

- `birdsong_fsk_sweeps.py`
- `birdsong_8band.py`

Archive as superseded, duplicate, or historical:

- `birdsong_bitmap.py`
- `birdsong_sweeps.py`
- `birdsong_sweeps_4sym.py`
- `birdsong_sweeps_8sym.py`
- `birdsong_2band.py`
- `birdsong_4band.py`
- `poc.py`
- challenge/coursework-oriented `modem.py`
- current `challenge/` tree
- `DEV_NOTES.md`

## Implementation Goals

1. Recenter the repo on `birdsong.py`

- Make `just send`, `just recv`, and the primary end-to-end recipe use
  `birdsong.py`.
- Add a real `just test` command.
- Make the README match the actual supported commands and file layout.

2. Make support status explicit

- Rewrite the README around three tiers: supported core, active experiments, and
  archived history.
- Remove unsupported certainty claims such as "100% reliable" unless a
  repeatable validation command backs them.
- Remove dead references such as `/workspace/`.

3. Clean the maintenance surface

- Replace `test_bit_conversion.py` with tests that exercise code that still
  exists.
- Update `.gitignore` to cover current generated byproducts such as `test_out/`
  and `_wav/`.
- Replace placeholder metadata in `pyproject.toml` with real project
  description.

4. Update project memory

- Record the support/archive policy in `docs/project_notes/decisions.md`.
- Record the stale-validation drift in `docs/project_notes/bugs.md`.
- Rewrite `docs/project_notes/key_facts.md` so it reflects the actual supported
  commands, files, and current assumptions.

5. Preserve the current engineering style

- Do not introduce a package split or framework-style abstraction layer just to
  organize the repo.
- Keep DSP code mostly as standalone scripts and focused helper functions.
- Preserve fade-in/fade-out behavior anywhere signal generation is kept.

## Testing and Acceptance

Testing approach:

- Use stdlib `unittest` in `tests/`.
- Add direct coverage for `birdsong.py` bit conversion, checksum, note parsing,
  and tone generation invariants.
- Add subprocess smoke tests for:
  - file round-trip through `birdsong.py`
  - pipe round-trip through `birdsong.py`
  - one file-based smoke test each for `birdsong_fsk_sweeps.py` and
    `birdsong_8band.py`

Acceptance criteria:

- `just check`, `just test`, and the primary end-to-end command pass.
- Root commands and README agree exactly.
- No broken tests remain for removed or archived code.
- The repo root contains only supported surfaces and no ambiguous historical
  entrypoints.

## Assumptions

- This is a research repo, not a packaging/distribution exercise.
- "Active experiment" means visible and smoke-tested, not production-grade.
- Challenge cleanup is part of the restructure, with room for a second pass that
  further improves the archived challenge history after the main migration lands.
